
### AI가전 개발 데이터 확보 및 공동활용 지원

홈페이지 : EC2

데이터 검색 및 필터링

메타데이터 저장 : Aurora DB

대용량 학습 데이터 저장 : S3

다운로드 : S3 Pre-signed URLs

데이터 업로드 및 전처리 : Lambda or batch

사용자 인증 및 권한 관리 :  Amazon Cognito / AWS IAM

####  데이터를 구매해서 대용량의 데이터를 어떻게 업로드하는게 제일 효율적인가?
1. **AWS CLI (Command Line Interface) 또는 AWS SDK:**
    
    - **가장 일반적이고 유연한 방법입니다.** aws s3 cp 명령어나 aws s3 sync 명령어를 사용하면 됩니다.
        
    - **멀티파트 업로드(Multipart Upload):** AWS CLI 및 SDK는 대용량 파일을 자동으로 작은 조각(part)으로 나누어 병렬로 업로드합니다. 이 방식은 네트워크 오류 시 재시작이 쉽고, 대용량 파일 업로드 시간을 크게 단축시킵니다. GB에서 TB 단위 파일에 매우 효율적입니다.
        
    - **aws s3 sync:** 로컬 디렉토리와 S3 버킷을 동기화하여 변경된 파일만 업로드하거나 삭제된 파일을 반영할 때 유용합니다.
        
    - **장점:** 스크립트화 용이, 자동 멀티파트 업로드, 높은 신뢰성.
        
    - **단점:** 직접 관리해야 하는 서버(EC2 또는 온프레미스)가 필요.
        
2. **AWS Transfer Family:**
    
    - **기능:** SFTP, FTPS, FTP 프로토콜을 사용하여 S3 버킷으로 파일을 직접 업로드하고 다운로드할 수 있는 완전 관리형 서비스입니다.
        
    - **장점:** 기존 SFTP/FTP 기반 시스템과 쉽게 통합 가능, 서버 관리 불필요, 대용량 파일 전송에 최적화.
        
    - **단점:** CLI/SDK보다 비용이 더 발생할 수 있음.
        
3. **AWS Direct Connect 또는 AWS VPN:**
    
    - **기능:** 온프레미스 데이터센터와 AWS 간에 전용 네트워크 연결(Direct Connect) 또는 암호화된 터널(VPN)을 설정하여 안정적이고 빠른 대용량 데이터 전송을 가능하게 합니다.
        
    - **장점:** 매우 안정적인 고대역폭 연결, 예측 가능한 성능.
        
    - **단점:** 설정 비용 및 시간이 많이 소요.
        
4. **AWS Snow Family (매우 대용량 - 페타바이트급):**
    
    - **기능:** 물리적인 스토리지 디바이스(Snowball, Snowball Edge, Snowmobile)를 AWS에서 보내주면, 데이터를 로컬에서 디바이스에 복사한 후 다시 AWS로 보내 S3에 업로드하는 서비스입니다. 네트워크 전송이 비효율적일 정도의 초대용량(수십 TB ~ 페타바이트) 데이터에 적합합니다.
        
    - **장점:** 네트워크 대역폭 제약 없이 초대용량 데이터 전송 가능.
        
    - **단점:** 물리적 배송 시간 소요, 대용량 데이터에만 효율적.
        

**효율성 추천:** 대부분의 GB~TB 단위 데이터 업로드에는 **AWS CLI 또는 SDK를 사용한 멀티파트 업로드**가 가장 효율적이고 비용 효과적인 방법입니다.

####  데이터를 업로드하면서 DB에 메타데이터를 어떻게 저장하는가?

1. **S3 이벤트 알림 (Lambda 트리거):**
    
    - **가장 일반적이고 권장되는 방법입니다.**
        
    - **프로세스:**
        
        1. 데이터 파일을 S3 버킷에 업로드합니다. (위 질문 1의 방법 중 하나 사용)
            
        2. S3 버킷에 **"객체 생성" 이벤트 알림**을 구성합니다. (예: s3:ObjectCreated:* 이벤트)
            
        3. 이 이벤트가 발생하면 미리 정의된 **AWS Lambda 함수가 자동으로 트리거**됩니다.
            
        4. Lambda 함수는 새로 업로드된 S3 객체의 정보(버킷 이름, 객체 키/경로, 파일 크기 등)를 받습니다.
            
        5. Lambda 함수는 **두 가지 작업**을 수행할 수 있습니다:
            
            - **메타데이터 추출:** 만약 파일 자체에 메타데이터가 포함되어 있다면 (예: 이미지의 EXIF, 영상 파일의 내장 정보), Lambda 함수는 이 파일을 읽어 필요한 메타데이터를 추출합니다.
                
            - **별도 메타데이터 파일 처리:** 또는, 데이터 파일과 함께 해당 데이터에 대한 메타데이터 파일(예: JSON, CSV)을 업로드하도록 규칙을 만들고, Lambda 함수가 이 메타데이터 파일을 읽어 처리할 수도 있습니다.
                
        6. 추출된 메타데이터(및 S3 객체 URL)를 **Aurora 또는 DynamoDB에 저장**합니다.
            
2. **AWS Step Functions를 사용한 오케스트레이션:**
    
    - 더 복잡한 전처리/메타데이터 추출 워크플로우가 필요한 경우 유용합니다.
        
    - S3 이벤트 -> Lambda 트리거 -> Step Functions 시작 -> 여러 Lambda 함수 또는 AWS Batch 작업을 순차적으로/병렬로 실행하여 메타데이터 추출, 전처리, DB 저장 등의 복잡한 작업을 오케스트레이션합니다.

#### Lamba 주의사항
 Lambda 함수의 기본 실행 시간 제한은 3초이며, 최대 15분까지 늘릴 수 있습니다. 메타데이터 추출 및 DB 저장 작업이 이 시간 내에 완료되도록 함수를 최적화해야함.

## 시나리오 (1)

스노우볼 → S3 적재 → Lambda 트리거 → 메타데이터 DB 저장 → 조건검색/다운로드 제공

### 대량 데이터 다운로드 시나리오
 *2안으로 진행하지만 실시간도 필요하기 때문에 AWS Batch 활용 추천*
#### 1안) 실시간/온디맨드 압축 : 
1. 검색 및 대량 다운로드 요청
2. 
	A)
	(Lambda 내 스트리밍 압축): Lambda 함수는 S3에서 각 s3_object_key에 해당하는 파일을 스트리밍으로 읽어와 메모리 내에서 압축 (예: ZIP 파일)한 후, 압축된 데이터를 다시 S3에 임시 파일로 저장합니다. (Lambda의 메모리 및 실행 시간 제약으로 인해 매우 큰 데이터셋에는 한계가 있습니다.)

	B)
	 (AWS Batch/EC2 사용): 데이터 양이 매우 많아 Lambda의 한계를 넘어서는 경우, Lambda는 AWS Batch 작업 또는 EC2 인스턴스를 시작하여 실제 압축 작업을 수행하도록 트리거합니다.
3.   압축된 임시 ZIP 파일이 S3에 저장되면, Lambda (또는 Batch/EC2 작업이 완료 시 트리거되는 별도 Lambda)는 이 임시 ZIP 파일에 대한 Pre-signed URL을 생성합니다.

	문제점 및 고려사항
		1. 압축 작업에 시간이 오래 걸릴 수 있어,(특히 수백 GB 이상). 사용자가 기다려야 함
		2.  Lambda의 한계를 넘어서면 AWS Batch나 EC2를 사용해야 하므로 아키텍처가 복잡해지
		   고 비용이 증가
		3. 압축된 파일을 저장할 임시 S3 버킷 관리 정책이 필요
		4. 압축 작업이 실패했을 경우의 재시도 및 사용자 알림 로직이 필요

#### 2안) 사전 정의된/자주 요청되는 데이터셋을 미리 압축하여 제공
1. 자주 사용될 것으로 예상되는 조건의 데이터셋을 미리 정의
2.  WS Batch 또는 EC2 인스턴스에서 압축
3. 압축된 결과 파일을 새로운 S3 객체로 저장
4. 소규모는 lambda의 메모리 내에서 ZIP으로 압축해서 S3에 저장
	- Lambda : 최대 15분, 10GB 메모리
