### AI 가전 개발 데이터 플랫폼

---
#### **1. 목표**
*   AI 가전 개발 데이터 확보/활용 플랫폼 구축
*   **핵심:** 대용량 데이터 업로드/수집, 메타데이터 관리, 통합 다운로드 전략 효율화

#### **2. 핵심 AWS 서비스**
*   **홈페이지/API:** `EC2`
*   **메타데이터 DB:** `Aurora DB`
*   **대용량 학습 데이터 저장:** `S3`
*   **다운로드 링크:** `S3 Pre-signed URLs`
*   **경량 작업/트리거:** `Lambda`
*   **대용량 배치 작업:** `AWS Batch`
*   **인증/권한:** 
		`AWS IAM`: **리소스** (S3, Lambda, EC2, Aurora 등)에 접근하는 사람, 애플리케이션, 서비스.
		`Amazon Cognito`: 애플리케이션의 최종 사용자를 위한 인증 및 권한 관리.

---

#### **3. 데이터 수집/업로드 전략 (어떻게 S3로 데이터를 가져올까?) **

**✅ 권장 방식: 벤더로부터 AWS S3로 직접 수신**
*   **상황:** 데이터 벤더가 AWS 클라우드를 사용하거나, 대량 데이터 전송 효율이 중요할 때.
*   **방법 1: S3 Direct Copy (Cross-account)**
    *   벤더의 S3 버킷에서 우리 S3 버킷으로 직접 데이터 복사. 가장 빠르고 효율적.
    *   `IAM 역할` 및 `버킷 정책`으로 안전하게 권한 부여.
*   **방법 2: AWS Snow Family (물리적 전송)**
    *   **대상:** 페타바이트(PB)급 초대용량 데이터 (네트워크 전송 비효율적일 때).
    *   **과정:**
        1.  AWS가 Snowball/Snowball Edge/Snowmobile 장비를 벤더에게 배송.
        2.  벤더가 로컬 데이터를 장비에 복사.
        3.  장비 재포장 후 AWS로 반송.
        4.  AWS가 장비 데이터를 우리 S3 버킷으로 업로드.
    *   **장점:** 네트워크 대역폭 한계 없이 초대용량 데이터 전송 가능.
    *   **`AWS Snow Family` 간단 정리:**
        *   **`Snowball`:** 기본 데이터 전송 장비 (50-80TB). 컴퓨팅 없음.
        *   **`Snowball Edge`:** Snowball + EC2 인스턴스처럼 컴퓨팅 기능 추가. 전처리/엣지 컴퓨팅 가능. (100TB)
        *   **`Snowmobile`:** 컨테이너 트럭 형태. 수십~수백 PB급 초대용량 전송.

**⚠️ 비권장 방식: 로컬 시스템을 통한 업로드 (최후의 수단)**
*   **상황:** 벤더가 AWS로 직접 전송하는 것이 불가능하고, 데이터를 우리 로컬 서버에서 직접 받아 S3로 다시 올려야 할 때.
*   **방법 1: `AWS CLI` or `AWS SDK`**
    *   `aws s3 cp` / `aws s3 sync` 명령어 사용.
    *   자동 `멀티파트 업로드` 지원 → 큰 파일 쪼개서 병렬 전송 (빠르고 안정적).
*   **방법 2: `AWS Transfer Family`**
    *   기존 SFTP/FTPS/FTP 기반 시스템 연동 시 활용.
    *   완전 관리형이라 서버 직접 관리 불필요.
*   **방법 3: `AWS Direct Connect` or `AWS VPN`**
    *   온프레미스 ↔ AWS 간 안정적이고 빠른 연결이 필요할 때.

#### **4. 메타데이터 저장 (데이터 S3 적재 후 DB에 정보 남기기) **

*   **자동화 핵심 1: `S3 이벤트 알림` → `Lambda 트리거` (데이터 파일 자체 메타데이터 추출)**
    1.  **S3에 데이터 파일 업로드**
    2.  **S3:** "새 객체 생성" 이벤트 발생
    3.  **Lambda:** 자동 실행 → S3 객체 정보 가져오기
    4.  **Lambda 기능:** 파일 자체에서 메타데이터 추출 (EXIF 등) → 추출 데이터 + S3 객체 URL을 `Aurora DB`에 저장.
    *   **주의:** Lambda 15분, 10GB 메모리 제한 내 최적화 필수.

*   **자동화 핵심 2: 벤더 제공 `매니페스트 파일` 활용 (강력 권장!)**
    1.  **벤더:** 데이터와 함께 `매니페스트 파일` (JSON, CSV 등)을 `S3`에 업로드
    2.  **S3:** "매니페스트 파일 생성" 이벤트 발생
    3.  **Lambda:** 자동 실행 → 매니페스트 파일 내용 읽기
    4.  **Lambda 기능:**
        *   매니페스트 내 **각 데이터 파일 정보(메타데이터)** 추출.
        *   매니페스트 내 **`고유값`을 `S3 객체 키`와 동기화(매핑)** 하여 DB에 저장.
    *   **장점:** 대량 메타데이터 효율적 일괄 처리, 일관성, 파싱 로직 간결.
    *   **주의:** 매니페스트 형식/S3 객체 키 매핑 규칙 명확히 정의.

*   **복잡한 워크플로우:** `AWS Step Functions`로 여러 Lambda/Batch 오케스트레이션.

#### **5. 통합 다운로드 전략 (대용량 데이터 어떻게 내려받게 할까?) **

**목표:** 사용자 요청 데이터 크기/빈도에 따라 최적의 다운로드 방법 제공

1.  **사전 압축 데이터셋 (Pre-compressed Datasets)**
    *   **대상:** `여자 30대 파일 전부` 등 **크고, 요청 빈도 높으며, 명확히 정의된** 데이터셋.
    *   **처리:** 벤더에게 요청 (가장 좋음!) **OR** `AWS Batch`로 주기적/업데이트 시 미리 압축 → `S3` 저장 (DB에 등록!).
    *   **장점:** **가장 빠르고 안정적**인 다운로드, 시스템 부하/비용 최소화.
    *   **단점:** 모든 조합 미리 압축 불가.

2.  **소규모 온디맨드(On-Demand) 압축 (`AWS Lambda` 활용)**
    *   **대상:** 검색 후 **비교적 적은 파일** (예: 총 용량 10GB 이내, 파일 50개 이내).
    *   **처리:** `Lambda` → S3 파일 **스트리밍 읽기 → 메모리에서 ZIP 압축** → S3 임시 업로드 → `Pre-signed URL` 반환.
    *   **장점:** **즉각적 응답**, 서버리스 (관리/비용 효율), 좋은 사용자 경험.
    *   **단점:** `Lambda` 실행 시간/메모리 제약으로 처리량 한계.

3.  **대용량 온디맨드(On-Demand) 압축 (`AWS Batch` 활용)**
    *   **대상:** **많은 파일** (예: 총 용량 10GB 초과, 파일 50개 초과) 또는 사전 압축되지 않은 대용량 조건 요청 시.
    *   **처리:** API Gateway → `Lambda` (트리거) → `AWS Batch` 작업 제출.
        *   `AWS Batch`가 S3 원본 다운로드 → 압축 → S3 임시 업로드.
        *   완료 후: `Lambda`가 `SNS`/이메일 알림 **OR** "다운로드 현황" 페이지에서 `Pre-signed URL` 제공.
    *   **장점:** `Lambda` 한계 넘어선 **대용량 압축 처리 가능**, `스팟 인스턴스` 활용으로 **비용 효율적**, 안정적인 비동기 처리.
    *   **단점:** 사용자 대기 시간 필요, 아키텍처 복잡성, 작업 상태 관리 필요.
