## 원인

### 핵심 오류 로그

```
Mar 25 09:00:26 localhost kernel: Cannot get hvm parameter CONSOLE_EVTCHN (18): -22!
Mar 25 09:00:26 localhost kernel: Cannot get hvm parameter CONSOLE_EVTCHN (18): -22!
```

### 원인 설명

- t2.large는 구형 **젠(Xen)** 하이퍼바이저 기반
- 부팅 시 커널이 Xen에게 콘솔 채널 번호(CONSOLE_EVTCHN)를 요청
- Xen이 해당 파라미터를 지원하지 못해 `-22 (EINVAL)` 반환
- 평소에는 커널이 오류를 무시하고 부팅 계속 진행
- 간헐적으로 타이밍 이슈로 부팅이 중단됨
- 매일 재부팅하는 환경에서 확률적으로 발생

### 부팅 이력

```
Mar 16 ~ Mar 25  매일 09:00 정상 부팅 ✅
Mar 26 09:02     부팅 실패 (1분만에 종료) ❌
Mar 26 10:04     재부팅 후 정상 복구 ✅
```

## 해결

### 임시 해결

- 수동 재부팅으로 정상 복구

### 근본 해결 (권고) - t2 → t3 변경

```
1. 인스턴스 중지 (Stop)
2. 인스턴스 유형 변경 → t3.large 선택
3. 인스턴스 시작
```

|항목|t2.large|t3.large|
|---|---|---|
|하이퍼바이저|젠 Xen (구형)|나이트로 Nitro (최신)|
|직렬 콘솔|❌ 미지원|✅ 지원|
|이 오류|재발 가능|발생 안 함|
|비용|상대적으로 높음|약간 저렴|

## 참고

- `CONSOLE_EVTCHN (18): -22` → 리눅스 에러코드 EINVAL, Xen이 콘솔 채널 파라미터를 모른다는 의미
- t2는 젠(Xen) 기반이라 EC2 직렬 콘솔 미지원 → 부팅 실패 시 복구가 어려움
- t3 이상(나이트로 Nitro 기반)은 직렬 콘솔 지원 → 부팅 실패 시 직접 접속해서 복구 가능
