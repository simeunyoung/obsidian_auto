
현재 외부 호출 구조는 다음과 같다.

`API Gateway → internal ALB(HTTPS listener) → Target Group(HTTP) → EC2`

문제는 내부 서비스 간 호출도 한동안 사실상 외부 호출과 유사하게 `EC2 → api.godamda.kr → API Gateway/ALB → Target Group → EC2` 경로를 타고 있었다는 점이다.

즉 같은 내부망 서비스 간 통신인데도, 직접 내부 경로를 사용하지 않고 외부 공개 진입점을 통해 다시 내부로 들어오는 구조였다.  
이 구조는 기능적으로는 동작할 수 있지만, 내부 통신 관점에서는 비효율적이고 장애 지점을 늘리는 방식이다.

---

## 문제 인식 계기

내부 호출이 외부 경로를 타는 구조에서 다음과 같은 timeout 에러가 발생했다.

.netty.channel.unix.Errors$NativeIoException: recvAddress(..) failed: Connection timed out  
WebClientRequestException: recvAddress(..) failed: Connection timed out  
Request to POST https://api.godamda.kr/api/1.0/device/execute

이 에러는 당시 **SSL 오류가 직접 원인으로 확인된 것은 아니고**,  
내부 호출이 외부 공개 도메인과 외부 진입 경로를 거치면서 **불필요한 네트워크 홉과 외부 구간 의존성이 생겼고, 그로 인해 timeout 가능성이 커진 것**으로 판단하였다.

즉, 이번 검토의 출발점은 다음과 같다.

- 내부 서비스 간 호출인데 외부 공개 도메인을 타고 있음
    
- 내부 호출이 외부 경로를 경유하면서 timeout 발생
    
- 내부 통신을 내부 전용 경로로 분리할 필요가 생김
    
- 이중화는 유지해야 하므로 LB를 활용한 내부 경로 재구성이 필요함
    

---

## 개선 목표

내부 호출을 다음 방향으로 바꾸는 것을 목표로 했다.

`EC2 → 내부용 Load Balancer → Target Group → EC2`

즉,

- 외부 호출은 기존처럼 API Gateway 경유
    
- 내부 호출은 외부 공개 진입점이 아니라 내부 전용 경로 사용
    
- 이중화는 유지
    
- 내부 호출의 불필요한 외부 경유 제거
    

---

## 검토한 방안

### 1. internal NLB

구조:

`EC2 → internal NLB → Target Group → EC2`

이 방식은 내부 호출을 TCP 레벨에서 단순 전달하는 구조다.  
애플리케이션은 실제로 HTTP 요청을 보내더라도, NLB는 내용을 해석하지 않고 포트 기준으로 그대로 전달한다.

#### 장점

- 구조가 단순함
    
- 내부 이중화 목적에 적합함
    
- 외부 공개 경로를 제거할 수 있음
    
- 내부 통신 전용으로 빠르게 적용 가능
    
- SSL 이름 정합성 같은 고민 없이 구성하기 쉬움
    

#### 단점

- Path 기반 라우팅 불가
    
- Host/Header 기반 제어 불가
    
- HTTP API 운영 관점의 가시성이 ALB보다 약함
    
- 서비스가 늘어나면 포트 관리가 복잡해질 수 있음
    

#### 적합한 경우

- 내부 호출 목적이 단순 이중화일 때
    
- 포트 기반 서비스 구분이면 충분할 때
    
- 내부에서 HTTP 기능을 LB가 이해할 필요가 없을 때
    

---

### 2. internal ALB + Route 53 private DNS

구조:

`EC2 → internal ALB → Target Group → EC2`

단, 호출 주소는 ELB 기본 주소가 아니라 private DNS 이름을 사용한다.

예:

- `dcm.internal.xxx`
    
- `fpn.internal.xxx`
    

#### 장점

- 내부 HTTP API 구조와 가장 잘 맞음
    
- 내부 서비스 호출을 도메인 기반으로 정리 가능
    
- 장기적으로 path/host 기반 확장이 가능함
    
- 운영 구조가 가장 정돈됨
    

#### 단점

- private DNS 구성 필요
    
- NLB보다 설계와 운영이 조금 더 복잡함
    
- 내부 도메인 운영 정책이 필요함
    

#### 적합한 경우

- 내부 호출이 REST API 중심일 때
    
- 향후 서비스가 늘어나거나 구조가 복잡해질 가능성이 있을 때
    
- 내부 호출도 HTTP 서비스 기준으로 체계화하고 싶을 때
    

---

### 3. internal ALB + 도메인 등록(public DNS 또는 외부 DNS CNAME)

구조 자체는 가능하다.  
외부 DNS에 CNAME을 등록하여 internal ALB 이름으로 연결하는 방식이다.

#### 장점

- 기존 DNS 운영 방식과 유사하게 적용 가능
    
- 도메인 이름으로 호출 가능
    

#### 단점

- 내부 전용 서비스명을 public DNS에 노출하게 됨
    
- 내부 전용 구조로는 private DNS보다 덜 깔끔함
    
- 설계상 내부용과 외부용 이름 체계가 섞일 수 있음
    

#### 적합한 경우

- private DNS를 바로 쓰기 어려운 환경
    
- 과도기적으로 빠르게 도메인 기반 호출을 맞춰야 할 때
    

---

## 방안 비교

|방안|장점|단점|적합한 상황|
|---|---|---|---|
|internal NLB|단순, 빠름, 내부 이중화 목적에 적합|HTTP 라우팅 불가, 포트 관리 복잡 가능|내부 단순 이중화|
|internal ALB + private DNS|내부 HTTP API 구조와 가장 잘 맞음, 확장성 좋음|구성 다소 복잡|REST API 중심, 장기 구조화|
|internal ALB + 도메인 등록|도메인 기반 호출 가능|내부 전용 이름 관리 관점에서 덜 깔끔|과도기적 적용|

---

## 판단 기준

이번 검토에서 핵심은 **SSL 오류 자체가 처음부터 확정 원인이었던 것이 아니라**,  
내부 호출이 외부 공개 경로를 타면서 timeout이 발생했고, 그 구조가 개선 여지가 있는 방식이라는 점이었다.

즉 판단 기준은 다음과 같았다.

1. 내부 호출이 외부 경로를 타지 않게 할 것
    
2. 이중화는 유지할 것
    
3. 내부 서비스 간 호출 경로를 단순화할 것
    
4. 현재 구조와 운영 난이도를 함께 고려할 것
    

---

## 결론

이번 검토를 통해 내부 호출은 외부 공개 진입점(`api.godamda.kr`)을 재사용하지 않고, 내부 전용 경로로 분리하는 것이 맞다고 판단했다.

정리하면:

- **문제의 출발점**은 내부 호출이 외부 경로를 타면서 timeout이 발생한 것
    
- **SSL 이슈**는 이후 내부 LB를 어떤 방식으로 구성할지 검토하는 과정에서 함께 고려하게 된 요소
    
- **단기적으로 가장 실용적인 방안**은 `internal NLB`
    
- **장기적으로 가장 정돈된 구조**는 `internal ALB + private DNS`

## 최종 정리

현재 상황에서는 다음처럼 이해할 수 있다.

### 외부 호출

`API Gateway → internal ALB → Target Group → EC2`

### 내부 호출(개선 전)

`EC2 → api.godamda.kr → API Gateway/ALB → Target Group → EC2`

### 내부 호출(개선 방향)

`EC2 → internal LB → Target Group → EC2`

즉,  
내부 호출을 외부 공개 경로에서 분리하여 내부 전용 경로로 전환하는 것이 이번 검토의 핵심이었다.

---

## 내가 정리한 해결 방향

이번에는 내부 호출이 외부 공개 도메인을 타는 구조 자체가 비효율적이고 timeout 가능성을 높인다고 판단하여, 내부 전용 LB 경로로 바꾸는 방향으로 검토를 진행했다.

- 빠르게 안정화하려면 `internal NLB`
    
- 구조를 장기적으로 정리하려면 `internal ALB + private DNS`
    

내 기준으로는 **지금 당장 현실적인 해결책은 internal NLB**,  
**장기적인 베스트 구조는 internal ALB + private DNS**로 정리할 수 있다.
