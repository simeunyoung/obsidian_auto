
# Spring Boot 로그 구조 개선기

### 서비스 로그 남발에서 Filter + Interceptor 기반 표준 로깅으로

## 1. 기존 로그 구조의 문제점

기존에는 대부분의 로그를 **Service 단위에서 직접 `log.info()`로 출력**하고 있었다.

`log.info("Received user info request: {}", userRequest); log.info("Found user ID for info request: {}", userId);`

이 방식은 초기에는 빠르게 개발하기 좋지만, 실제 운영 환경에서는 여러 문제를 드러냈다.

### ❌ 문제점

- **요청 단위 로그 흐름이 보이지 않음**
    
    - 요청 시작 / 종료 시점이 명확하지 않음
        
- **로그 중복**
    
    - 같은 요청에 대해 Controller, Service에서 유사한 로그가 반복
        
- **비즈니스 코드 오염**
    
    - 로그 코드가 서비스 로직 가독성을 해침
        
- **Request / Response 전체 흐름 파악 어려움**
    
    - 요청 바디, 응답 바디, 처리 시간 등을 한눈에 보기 어려움
        

---

## 2. 로그 구조 개선 목표

로그 구조를 개선하면서 다음 원칙을 세웠다.

### ✅ 목표

1. **요청 단위 로그를 한 곳에서 관리**
    
2. **비즈니스 로직과 로깅 로직 분리**
    
3. **운영 로그와 디버그 로그 명확히 구분**
    
4. **필요할 때만 Request / Response Body 확인 가능**
    

---

## 3. 개선된 로그 아키텍처 개요

개선 후 로그 구조는 다음과 같다.

`[Filter]  └─ Request / Response Body 캐싱  └─ DEBUG 레벨에서만 Body 로그 출력  [Interceptor]  └─ 요청/응답 메타 정보 로그 (INFO)  [Service]  └─ 비즈니스 상태 로그만 출력`

각 계층의 역할을 명확히 분리했다.

---

## 4. CacheBodyFilter – Body를 읽기 위한 기반 작업

### 왜 Filter가 필요한가?

HTTP Request/Response Body는 **기본적으로 한 번만 읽을 수 있다.**

- Controller에서 읽으면 Interceptor에서는 읽을 수 없음
    
- Interceptor에서 읽으면 Controller 로직이 깨짐
    

이를 해결하기 위해 `ContentCachingRequestWrapper`,  
`ContentCachingResponseWrapper`를 사용하는 Filter를 추가했다.

### CacheBodyFilter 역할

`@Component public class CacheBodyFilter extends OncePerRequestFilter {      @Override     protected void doFilterInternal(             HttpServletRequest request,             HttpServletResponse response,             FilterChain filterChain     ) throws ServletException, IOException {          ContentCachingRequestWrapper requestWrapper =                 new ContentCachingRequestWrapper(request);         ContentCachingResponseWrapper responseWrapper =                 new ContentCachingResponseWrapper(response);          try {             filterChain.doFilter(requestWrapper, responseWrapper);         } finally {             responseWrapper.copyBodyToResponse();         }     } }`

#### 핵심 역할

- Request / Response Body를 **메모리에 캐싱**
    
- 이후 Interceptor 또는 Filter에서 안전하게 Body 접근 가능
    
- `copyBodyToResponse()` 호출로 응답 손실 방지

## 5. LoggingInterceptor – 운영 로그의 중심

### Interceptor에서 담당하는 로그

Interceptor에서는 **Body를 찍지 않는다.**

운영 환경에서 항상 필요한 **메타 정보 로그**만 남긴다.

#### preHandle (요청 시작)

`log.info("[REQ] {} {} | IP={}",         request.getMethod(),         request.getRequestURI(),         request.getRemoteAddr() );`

- HTTP Method
    
- URI
    
- Client IP
    
- traceId(MDC) 생성
    

#### afterCompletion (요청 종료)

`log.info("[RES] {} {} | status={} | duration={}ms",         request.getMethod(),         request.getRequestURI(),         response.getStatus(),         duration );`

- 응답 상태 코드
    
- 처리 시간(ms)
    
- 예외 발생 시 에러 로그
    

### Interceptor를 사용하는 이유

- Controller 기준 전/후 시점이 명확
    
- 비즈니스 로직과 완전히 분리
    
- 운영 로그로서 안정적이고 가벼움
    

---

## 6. Request / Response Body 로그는 DEBUG에서만

### 왜 Body 로그는 항상 찍지 않는가?

- 개인정보(비밀번호, 토큰) 포함 가능
    
- 로그 용량 급증
    
- 직렬화 및 IO 비용 증가
    
- 운영 로그 가독성 저하
    

### 해결 방식

- **Filter 레벨에서**
    
- **DEBUG 로그일 때만**
    
- Request / Response Body 출력
    

`if (log.isDebugEnabled()) {     log.debug("[REQ BODY] {}", requestBody);     log.debug("[RES BODY] {}", responseBody); }`

👉 운영 환경에서는 INFO 로그만 확인  
👉 문제 발생 시 DEBUG 활성화로 상세 분석 가능

## 7. 서비스(Service) 로그는 어떻게 바뀌었나

### ❌ 제거한 로그

`log.info("Received request ..."); // 인터셉터가 이미 담당`

### ✅ 남긴 로그

`log.debug("password validation start"); log.info("userId={} groupCount={}", userId, groupList.size()); log.warn("invalid password for userId={}", userId); log.error("user info processing failed", e);`

**비즈니스 의미가 있는 로그만 남기도록 정리**했다.

---

## 8. 개선 효과

### 로그 개선 전 vs 후

|항목|개선 전|개선 후|
|---|---|---|
|요청 추적|어려움|traceId 기준 명확|
|로그 위치|Service 난립|Filter / Interceptor 분리|
|Body 로그|무분별|DEBUG 전용|
|운영 안정성|낮음|높음|
|가독성|낮음|높음|

---

## 9. 마무리

이번 로그 구조 개선을 통해 다음을 얻었다.

- 요청 단위 로그 흐름이 한눈에 보임
    
- 비즈니스 코드가 훨씬 깔끔해짐
    
- 운영/디버깅 로그 전략이 명확해짐
    
- Spring Boot 표준에 가까운 구조 확보
    

**“로그는 많을수록 좋은 것이 아니라, 역할이 명확해야 한다”**는 점을 다시 한 번 느낀 작업이었다.
