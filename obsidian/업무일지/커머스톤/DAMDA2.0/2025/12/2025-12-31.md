### OTA

현재 OTA Lambda 구조는 작업을 스냅샷 방식으로 운영하도록 설계되어 있음.
스냅샷 -> 연속 방식으로 변경함에 따라 로직 수정이 필요한 상태이며,
현재는 임시 운영을 위해 일부 로직을 비활성화하여 운용 중임.
향후 정식 구조로의 개선이 필요함.

---

현재 운영 로직
	1. IoTCore에 OTA 작업이 생성되면 
		- `iot_device_firmware` 테이블에 완료되지 않은(SUCCEEDD) 레코드가 있는지 확인
		- 조건 충족 시 `iot_model_firmware` 테이블에 completed = 1로 데이터 등록
	2. EventBridge(수요기업 기준 1분 주기) 
		- `iot_model_firmware` completed = 0 인 버전 조회
		- 해당 Job에 대해 IoTCore에 `list_job_executions_for_job` 호출
	3. 디바이스 실행 상태 동기화
		- JobExecution의 uuid로 serial 조회
		- `device_id = manufacturer-model-serial` 생성
		- iot_device_firmware에 
			- 데이터가 존재 시 -> UPDATE
			- 데이터가 미존재 시 -> INSERT
	4. 모델 단위 완료 여부 확인
		- SUCCEEDD가 아닌 디바이스가 존재하는지 확인
		- 모두 완료된 경우 
			- `iot_model_firmware`.completed = 1로 업데이트
	5. IoTCore OTA 작업 삭제 시
		- `iot_model_firmware` 데이터 삭제
	6. 디바이스 삭제 시 
		- iot_device_firmware 데이터 삭제

---

임시 운영 로직
	- 모든 디바이스가 완료된 경우에도
	`iot_model_firmware`.completed = 1로 변경하는 로직을 임시로 비활성화
	- completed = 0 상태를 유지하여
	  주기적인 JobExecution 추적이 가능하도록 운영 중

---

향후 개선 방향
- 현재 OTA 구조는 **스냅샷 기반 설계**를 전제로 하여  
연속 Job 운영 시 완료 판정(`completed`) 로직과 충돌이 발생함.  
이에 따라 OTA 판단 주체를 **Job 중심 → 디바이스 중심**으로 전환하는 개선이 필요함.

### 1. OTA 판단 기준을 디바이스 중심으로 전환

#### As-Is

- Job 단위로 전체 완료 여부(`completed`) 판단
- 신규 디바이스 등록 시 기존 Job과의 연계가 불명확
#### To-Be

- 디바이스 등록 시점에
    - 해당 디바이스의 현재 펌웨어/소프트웨어 버전 확인
    - 모델별 최신 OTA 버전과 비교
- **디바이스 기준으로 OTA 필요 여부 판단**

### 2. 디바이스 등록 시 OTA 트리거 구조 도입 (핵심)

#### 개선 방향

- 디바이스 등록(Thing 생성) 시:
    1. 서버에서 모델 기준 최신 OTA 버전 조회
    2. 디바이스 버전이 낮은 경우
        - **해당 디바이스 단독 스냅샷 Job 생성**
    3. Job 대상은 **단일 Thing**

#### 기대 효과

- 연속 Job 의존 제거
- 중고/재등록 디바이스 즉시 대응 가능
- OTA 실행 시점 명확화
